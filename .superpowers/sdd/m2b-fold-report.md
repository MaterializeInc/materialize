# M2b scalar nullability fold report

## What changed

### `interner.rs` — `intern_scalar_in_context`

Added `intern_scalar_in_context(expr, col_types)`.
The existing `intern_scalar` now delegates to it with an empty slice.

The method interns the *original* expression unchanged (round-trip identity
preserved) and separately folds a *clone* of the expression against the
supplied column types via `MirScalarExpr::reduce(col_types)`.
The fold result sets the `lit` flag:

* `Some(true)` if the folded clone `is_literal_true()`
* `Some(false)` if the folded clone `is_literal_false()`
* `None` otherwise

The stored scalar bytes are from the original; only the flag uses the fold.

### `lower.rs` — Filter and Map arms

Both arms now call `input.typ().column_types` before lowering the input,
then pass the result to a new `intern_all_in_context` helper that calls
`intern_scalar_in_context` for each scalar.

`input.typ()` is an O(n) tree traversal; we call it once per Filter/Map
node, not per scalar.

## How the IS NULL test predicate was built

`MirScalarExpr::column(0).call_is_null()` — `call_is_null()` wraps the
expression in `CallUnary { func: UnaryFunc::IsNull(IsNull), expr }`.
The `IsNull` struct is generated by the `#[sqlfunc]` macro in `datum.rs`.
After `reduce` over a NOT NULL column type, the expression folds to
`Datum::False`, making `is_literal_false()` return true.

## Tests

Three TDD tests added to `roundtrip.rs`:

* `is_null_on_not_null_folds_to_empty` — `IS NULL(#0)` on a NOT NULL column
  optimizes to an empty constant via `empty_false_filter`.
* `is_null_on_nullable_stays` — `IS NULL(#0)` on a nullable column does NOT
  become an empty constant.
* `column_ref_lit_is_unaffected` — bare column reference `#0` on NOT NULL
  column does not get `lit = Some(false)`; `empty_false_filter` must not fire.

All 41 tests pass (`cargo test -p mz-transform-egraph`).

## E-graph blowup and fix

Setting `lit = Some(false)` activated `any_false(p)` on filter predicates.
`distribute_filter_union_nary` and `merge_filters` then created unbounded new
predicate-list concat objects, each re-triggering `empty_false_filter` in a
positive feedback loop.

Fix: added `no_false(p)` condition to both distribution rules
(`distribute_filter_union` and `distribute_filter_union_nary`).
When a predicate is already known-false, `empty_false_filter` handles the
filter directly; distribution is unnecessary and must not fire.

`no_false` is a new `Cond` variant, parsed in `parser.rs`, evaluated in both
`egraph.rs` and `matcher.rs` (vacuously true on an empty scalar list).

A regression test `case5_filter_over_union_filter` covers the case that
previously caused the blowup.

## Concerns

None.
The no_false guard is conservative: it only blocks distribution when a
predicate is *provably* false, which is exactly when distribution is both
unnecessary and harmful to saturation termination.
