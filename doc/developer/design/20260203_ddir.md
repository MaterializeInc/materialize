## Problem

We take SQL queries and translate them into maintained dataflows.
The translation is layered into various abstraction levels, from a direct SQL representation to something suitable for rendering as a dataflow.
As we're getting further away from SQL, less of its intricacies matter, and we're getting closer to what we can render as a dataflow.

Each abstraction level works on an intermediate representation (IR):
* _Plans_ (sic) are a direct representation of SQL.
* _Hir_ are decorrelated (?) plans, with some SQL-isms left.
* _Mir_ is where the bulk of optimization work happens.
  It is meant to be a representation of what we can do with Differential collections.
* _Lir_ (the compute plan) are rendering instructions.
  It is mostly the same as Mir, but with additional information applied, such as single time, input monotonicity.
  The boundary between Lir operators are collection bundles, which can be arranged or raw collections.

However, Lir fails to precisely capture what dataflows we could render.
The design outlines what the problems are, what we could gain from adding another IR, and how such an IR might look like.

Dataflows are a graph of operators with edges to move data.
Each operator takes typed inputs, and produces various typed outputs.
(Differential says that the data has a certain structure.)
Lir defines operators on a high level, and Lir operators are backed by multiple Timely or Differential operators.

From Lir's perspective, the edges all have the same type:
* An edge carries a stream of data.
* An edge provides multiple arranged forms.
* One of the above must be true, i.e., there can't be an edge with neither a stream not an arranged form.

What we can't do right now:
* Prune error streams: They're an intrinsic part of collections, and both the stream and arranged form can represent errors.
* Efficient temporal filters: Arrangements are not efficient at maintaining future updates as we need to rescan data on every tick.
  We implemented a data structure to make scanning more efficient, but only want to use it in places where we might observe future updates.
* Pruning unused streams and arrangements: We always provide a full collection bundle, containing potentially more information than needed by the consuming operator.
  Providing the information comes at a cost, which we'd like to avoid.

## Solution Proposal

We introduce a new IR called Differential Dataflow IR (DDIR).
It represents a directed acyclic graph of operators with typed edges.
Recursion is modeled via `LetRec`, which allows cycles in the rendered dataflow while keeping the IR acyclic.

### Graph structure

```rust
struct Graph {
    nodes: Vec<Node>,
    edges: Vec<EdgeType>,
}

struct EdgeId(usize);

struct Node {
    op: Op,
    inputs: Vec<Vec<EdgeId>>,  // per logical input, multiple edges (unioned)
    outputs: Vec<EdgeId>,
}
```

An edge can serve as input to multiple nodes.
Each logical input can accept multiple edges, which are implicitly unioned.

### Edge types

```rust
enum EdgeType {
    Error,
    Collection,
    Arranged { key: Vec<MirScalarExpr>, val: Vec<usize>, row: Vec<usize> },
}
```

### Design principles

**Operators produce only what they create.**
Operators do not implicitly pass through inputs.
An `Mfp` node produces its own data and error outputs; it does not forward upstream errors.
Upstream edges remain available for downstream nodes to reference directly.

**Error collection is explicit.**
Use `Union` nodes to combine error edges at boundaries (Let bindings, before sinks):
```
n0: Get(t)           [] -> [e0: Collection, e1: Error]
n1: MfpFallible(f)   [[e0]] -> [e2: Collection, e3: Error]
n2: Union            [[e1, e3]] -> [e4: Error]
```

**Operators are basic and not subdividable.**
LIR's `Constant` (which produces `Result<rows, err>`) maps to two DDIR operators: `ConstantRows` and `ConstantErr`.

**Operators map to whole Differential/Timely operators.**
Each DDIR operator renders to one or more Differential/Timely operators, never to a partial operator.
Recovering fused operators from multiple DDIR nodes would be complex; instead, keep fusion at the DDIR level.

**Fallible vs infallible operators.**
Operators that may produce errors have fallible variants with an error output.
Infallible operators omit the error output:
```rust
MfpInfallible { mfp: MapFilterProject }  // [Collection] -> [Collection]
MfpFallible { mfp: MapFilterProject }    // [Collection] -> [Collection, Error]
```
During lowering, inspect the `MapFilterProject` to determine which variant to use.

### Operators

```rust
enum Op {
    // Constants
    ConstantRows { rows: Vec<(Row, T, Diff)> },
    ConstantErr { err: EvalError },

    // References
    Get { id: GlobalId },
    GetLocal { id: LocalId },  // references Let or LetRec binding

    // Bindings (LetRec supports mutual recursion)
    Let { id: LocalId },
    LetRec { ids: Vec<LocalId>, limits: Vec<Option<LetRecLimit>> },

    // Transforms
    MfpInfallible { mfp: MapFilterProject },
    MfpFallible { mfp: MapFilterProject },
    FlatMapInfallible { exprs: Vec<MirScalarExpr>, func: TableFunc, mfp_after: MapFilterProject },
    FlatMapFallible { exprs: Vec<MirScalarExpr>, func: TableFunc, mfp_after: MapFilterProject },

    // Relational
    Join { plan: JoinPlan },
    Reduce { key_val_plan: KeyValPlan, plan: ReducePlan, mfp_after: MapFilterProject },
    TopK { plan: TopKPlan },
    Threshold { plan: ThresholdPlan },

    // Structural
    Negate,
    Union,  // variadic, any edge type
    Arrange { key: Vec<MirScalarExpr>, val: Vec<usize>, row: Vec<usize> },
    Consolidate,
    Distinct,  // deduplicates (for error streams in recursion)

    // Recursion support
    IterationLimit { level: usize, max_iters: NonZeroU64 },
        // [Collection] -> [Collection, Collection] (in_limit, over_limit)
    LimitExceededErr,  // [Collection] -> [Error]

    // Regions (for visualization and dataflow structure)
    EnterRegion { name: String },
    LeaveRegion,
}
```

`Union` is variadic and associative: `union(union(a, b), c) = union(a, b, c)`.

### Inputs and outputs

Inputs: `Get`, `ConstantRows`, `ConstantErr`.

Outputs: terminal nodes or explicit capture operator (TBD).


## Mapping LIR operators

| LIR | DDIR |
|-----|------|
| `Constant { rows: Ok(data) }` | `ConstantRows { rows: data }` |
| `Constant { rows: Err(e) }` | `ConstantErr { err: e }` |
| `Get { id: Global(id), .. }` | `Get { id }` |
| `Get { id: Local(id), .. }` | `GetLocal { id }` |
| `Let { id, value, body }` | `Let { id }` with value and body subgraphs |
| `LetRec { ids, values, limits, body }` | `LetRec` with explicit internal operators (see below) |
| `Mfp` (infallible) | `MfpInfallible` |
| `Mfp` (fallible) | `MfpFallible` |
| `Union { inputs, consolidate_output: false }` | `Union` |
| `Union { inputs, consolidate_output: true }` | `Union` followed by `Consolidate` |
| `ArrangeBy` | `Arrange` (one per arrangement) |

Note: initial lowering produces all possible edges. A subsequent pruning pass removes unused edges.

### LetRec internal structure

`LetRec` in LIR renders to multiple Differential operators: `SemigroupVariable`, `consolidate`, `branch_when` (for limits), and `distinct` (for errors). To enable edge pruning, these are explicit DDIR operators.

Additional operators for recursion:
```rust
enum Op {
    // ... existing ...
    Distinct,  // deduplicates error stream to prevent unbounded accumulation
    IterationLimit { level: usize, max_iters: NonZeroU64 },
        // [Collection] -> [Collection, Collection] (in_limit, over_limit)
    LimitExceededErr,  // [Collection] -> [Error] (converts over_limit to error)
}
```

`LetRec` itself is structural: it defines scope boundaries and feedback points. The consolidation, limiting, and error handling are separate nodes:

```
LetRec { ids: [x] }
│
├── value subgraph for x:
│   │
│   │  ... computation referencing GetLocal(x) ...
│   │  e10: Collection (computation data)
│   │  e11: Error (computation errors, if fallible)
│   │
│   ├── Consolidate         [[e10]] -> [e12]
│   ├── IterationLimit      [[e12]] -> [e13, e14]  (in_limit, over_limit)
│   ├── LimitExceededErr    [[e14]] -> [e15]       (* only if !return_at_limit *)
│   ├── Union               [[e11, e15]] -> [e16]  (* combine errors *)
│   └── Distinct            [[e16]] -> [e17]       (* deduplicate errors *)
│
│   feedback: e13 (data), e17 (errors)
│
└── body subgraph
```

Pruning cases:
* **Infallible computation**: `e11` absent, `Union` simplifies, `Distinct` may be removed
* **`return_at_limit: true`**: `LimitExceededErr` absent, `e15` absent
* **No limit**: `IterationLimit`, `LimitExceededErr` absent; `e12` feeds directly to feedback
* **No errors**: entire error path pruned

### Regions

Timely regions group operators for visualization without changing timestamp semantics.
DDIR models these with `EnterRegion` and `LeaveRegion` operators.

```rust
EnterRegion { name: String }  // imports edges, starts region
LeaveRegion                   // variadic, exports all edges, ends region
```

Structure mirrors the code pattern `scope.region_named(name, |region| { ... })`:

```
e0: Collection (outer scope)

n1: EnterRegion { name: "Mfp" }   [[e0]] -> [e1]
n2: MfpFallible                   [[e1]] -> [e2, e3]
n3: LeaveRegion                   [[e2, e3]] -> [e4, e5]

(* e4, e5 available in outer scope *)
```

Properties:
* Both operators are identity on data; they affect only Timely's operator hierarchy
* `EnterRegion` imports edges into the region scope
* `LeaveRegion` is variadic: collects all region outputs and exports them, marking the region boundary
* Regions can nest
* Optional for correctness; useful for visualizing large dataflows

Note: `LeaveRegion` outputs are flat (`Vec<EdgeId>`), which does not allow us to distinguish different outputs.
Revisit if position-dependence for same-typed edges causes problems; grouped outputs (`Vec<Vec<EdgeId>>`) may be needed.

## Pruning unneeded edges

Traverse the graph backwards from outputs and remove edges not reachable from any output.

Simplifications:
* `Union([e])` → use `e` directly
* `Union([..., Union([a, b]), ...])` → `Union([..., a, b, ...])`
* Remove edges known to be empty (e.g., from infallible operators)
* Remove unused `Union` nodes

## Temporal filters

We introduce a `Temporal Bucketing` operator on edges that can have future updates and before arrangement-forming operators.
We can inspect MFPs to learn whether a certain edge can have future updates or not.

## Operator specialization

* An arrange-by that we only consume as a stream is just a consolidate operator.
  Not sure if this ever happens.
