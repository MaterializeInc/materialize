# mz-deploy: a CLI for developing and deploying Materialize projects

- Associated: (TBD — link the mz-deploy epic/PR)

## The Problem

Teams building on Materialize manage a growing body of SQL — views,
materialized views, sinks, and the clusters, sources, connections, and
secrets they run on — with no project tooling from Materialize itself. We
point users at dbt and Terraform. Each covers a slice of the problem.
Neither is an acceptable foundation.

### dbt

dbt-materialize is our documented answer for managing SQL as a project.

- **dbt is a data warehousing tool; Materialize is operational.** The
  mismatch shows up in the primary workflow. `dbt build` — the main way
  people use dbt — rebuilds models on a schedule. On Materialize,
  dataflows run continuously and rebuilding a materialized view means
  rehydrating it, so we tell users not to run `dbt build`. The tool's
  central command is wrong for our platform.
- **The vendor is moving away from us.** dbt Labs and Fivetran have
  merged, and the combined company is focused on warehousing. We are not
  onboarded to dbt Fusion (the new engine); there is no timeline. We are
  not onboarded to dbt Cloud; there is no timeline.
- **Deployment is bolted on.** Blue/green on dbt is managed via macros —
  user-space templating around a procedure the tool has no model of.
  There is no partial deployment (deploy only what changed). There is no
  way to support replacement views — updating a materialized view in
  place so downstream consumers, including other teams' projects, are
  untouched.
- **We do not control the primary interface to our product.** The project
  format, the CLI, the error messages, the deployment semantics — all
  owned by another vendor with different priorities. Every improvement we
  want must fit dbt's adapter seams or not happen.

### Terraform

The Materialize Terraform provider handles clusters, connections, and
secrets. That is all it handles. Terraform has no model of views,
materialized views, or indexes — where the development actually happens.
It cannot diff SQL logic. It cannot stage and swap a dependency graph.
Its state file duplicates what the catalog already knows.

### What's missing regardless of tool

Several needs have no answer in any existing tool:

- **No source of truth.** The SQL that defines an environment lives in
  ad-hoc scripts, migrations folders, or only in the catalog itself. There
  is no checked-in project that a team can review, diff, and redeploy.
- **Deployments are manual and risky.** Materialize's recommended
  zero-downtime pattern — build the new objects alongside the old, wait for
  hydration, swap, drop — is a documented manual procedure. Executing it by
  hand (or with macros and hand-rolled scripts) is error-prone, not
  resumable after a failure, and has no protection against two engineers
  deploying concurrently.
- **Errors surface too late.** Without a way to validate SQL against
  Materialize's planner offline, a typo'd column name or type error is
  discovered only when DDL is executed against the production environment.
- **No unit testing.** There is no way to test a view's logic against
  fixture data in CI without standing up sources and real upstream data.
- **Cross-team coupling.** When one team changes a materialized view that
  another team's project depends on, the blue/green pattern breaks the
  consumer: the producer's project cannot see, let alone redeploy, objects
  it doesn't own.
- **No editor support.** SQL files for Materialize get no completion,
  navigation, or diagnostics.

mz-deploy addresses all of these with one tool we own end to end: a project
model on disk, an offline compiler and typechecker, a unit test harness,
and a blue/green deployment lifecycle with atomic cutover, partial
deployments, replacement views, and crash recovery. An LSP server and VS
Code extension are built on the same compiler.

## Success Criteria

- A Materialize environment can be described as a checked-in directory of
  SQL files, and the tool can reconcile the environment to match it.
- Compilation and typechecking run entirely offline (no server connection)
  using Materialize's real parser and planner, so anything the server would
  reject at DDL time is caught before deployment; cost is proportional to
  what changed, not project size.
- Deployments are zero-downtime: production objects remain queryable
  throughout, and cutover is a single atomic transaction.
- Every deployment step is crash-safe: killing the tool at any point leaves
  the system in a state from which rerunning the same command resumes
  correctly, and no failure mode can destroy user data (tables, sources) or
  emit garbage to external systems (sinks).
- Concurrent use by multiple engineers is safe without an external lock
  service: concurrent stages don't collide, and conflicting promotes are
  detected and reported.
- A changed materialized view in a declared stable schema can be updated
  without redeploying — or even notifying — its downstream consumers, in
  this project or any other.
- View logic is unit-testable against inline fixture data, locally and in
  CI, with no remote environment.
- Editor features (diagnostics, completion, navigation, run-test/explain
  lenses) are answered by the same compiler the CLI uses, so editor and CLI
  can never disagree.
- Materialize owns the entire developer interface — project format, CLI,
  error messages, deployment semantics. Improvements ship on our timeline,
  not another vendor's.

## Out of Scope

- **Orchestration and scheduling.** mz-deploy is a CLI invoked by a human or
  a CI job; it does not schedule deployments, watch repositories, or run as
  a service.
- **Data migrations and backfills.** The tool manages object lifecycles. It
  never rewrites table data; altering or recreating a table is an explicit,
  human-confirmed `delete` followed by re-`apply`.
- **Managing objects outside the project.** Objects the project does not
  declare are invisible to the tool, with one deliberate exception: ad-hoc
  objects manually created inside a managed schema are dropped with it at
  promote time.
- **Process-level locking.** Safety under concurrency comes from the
  metadata design (disjoint per-deployment resources, duplicate-id
  rejection, promote-time conflict detection), not from a lock manager.

## Solution Proposal

### Summary

mz-deploy is a CLI for developing and deploying Materialize projects. A
project is a directory of SQL files that declare databases, schemas, views,
materialized views, sinks, and the infrastructure they run on (clusters,
roles, sources, connections, secrets). The CLI compiles that directory into
a dependency graph, validates it without a server, and deploys it to a
Materialize environment using a blue/green workflow with atomic cutover.

The design splits the world in two. The *logic* of a project — views and
materialized views — deploys blue/green: `stage` builds the changed subset
in suffixed schemas and clusters alongside production, `wait` watches the
new clusters hydrate, and `promote` swaps staging into production in one
transaction. The *infrastructure* — clusters, roles, secrets, connections,
sources, tables — is reconciled idempotently by the `apply` family and is
never swapped, because it is stateful. All deployment state lives in a
`_mz_deploy` database on the target server itself, so conflict detection
and crash recovery are plain SQL.

### The project model

```
project.toml
.mzprofile                    (optional, git-ignored)
types.lock                    (column types of external dependencies; see typechecking)
models/
  <database>/
    <database>.sql            (optional database-level statements)
    <schema>/
      <schema>.sql            (optional schema-level statements, e.g. ALTER SCHEMA ... SET api = stable)
      <object>.sql            (one object per file; path determines its FQN)
      <object>#<profile>.sql  (optional per-profile variant)
clusters/  roles/  network-policies/   (infrastructure definitions, SQL)
```

The file path is the object's identity: `models/db/schema/foo.sql` must
contain a CREATE statement for `db.schema.foo`, and the compiler rejects
mismatches. This convention matches standards of other languages like Java/Maven.

Each file kind accepts a closed set of statements, enforced at compile
time (`project/error/validation.rs` lists the rejections):

| File | Accepted statements |
|------|---------------------|
| `models/<db>/<schema>/<object>.sql` | One of `CREATE TABLE`, `CREATE TABLE ... FROM SOURCE`, `CREATE VIEW`, `CREATE MATERIALIZED VIEW`, `CREATE SOURCE`, `CREATE SINK`, `CREATE SECRET`, `CREATE CONNECTION` — plus companion `CREATE INDEX`, `GRANT`, `COMMENT ON`, and `EXECUTE UNIT TEST` statements for that object |
| `models/<db>/<schema>/<schema>.sql` | `COMMENT ON SCHEMA`, `GRANT ON SCHEMA`, `ALTER DEFAULT PRIVILEGES`, `ALTER SCHEMA ... SET api = stable` |
| `models/<db>/<db>.sql` | `COMMENT ON DATABASE`, `GRANT ON DATABASE`, `ALTER DEFAULT PRIVILEGES` |
| `clusters/*.sql` | `CREATE CLUSTER` plus grants/comments |
| `roles/*.sql` | `CREATE ROLE` plus grants |
| `network-policies/*.sql` | `CREATE NETWORK POLICY`, `GRANT`, `COMMENT ON` |

One structural rule cuts across all of this and is load-bearing for the
deployment design: within a schema, *storage* objects (tables, sinks) may
not mix with *computation* objects (views, materialized views) — see
[dependency analysis](#dependency-analysis) for why.

### Command surface and workflow

`src/bin/mz-deploy/main.rs` defines the clap CLI. Commands are grouped in
the help output the way users encounter them:

| Group | Commands |
|-------|----------|
| Getting started | `new`, `init`, `profile`, `setup`, `debug` |
| Develop | `compile`, `clean`, `test`, `explain`, `dev`, `lsp`, `sql`, `mcp` |
| Infrastructure | `lock`, `apply` (clusters, roles, network-policies, secrets, connections, sources, tables), `delete` |
| Deploy | `stage`, `wait`, `promote`, `abort`, `describe`, `list`, `log` |

Global flags: `--directory/-d` (project root), `--profile/-p` (connection
profile, also `MZ_DEPLOY_PROFILE`), `--profiles-dir` (defaults to `~/.mz`),
`--docker-image` (override the local Materialize image), `--output`
(`text` or `json`), `--verbose`, `--quiet`. Every server-mutating command —
`stage`, `promote`, `dev`, and each `apply` phase — supports `--dry-run`,
which plans and prints the SQL without executing it.

Each command is a module under `src/cli/commands/` exposing a
`run(...) -> Result<_, CliError>` function. Commands are stateless:
`main.rs` loads `Settings` once and passes it by reference. Commands that
only compile (`compile`, `test`, `explain`) work without a connection
profile; commands that talk to a server resolve one.

The intended workflow, end to end:

```
setup                 (once per environment: create _mz_deploy metadata + server cluster)
apply                 (idempotently reconcile clusters, roles, secrets, connections, sources, tables)
stage                 (deploy changed views/MVs into suffixed staging schemas + clusters)
wait <deploy-id>      (stream hydration status until staging clusters are ready)
promote <deploy-id>   (atomically swap staging into production, run deferred work)
```

`dev` is a separate scratch path for iterating against a remote
environment: it rebuilds a throwaway per-developer overlay database on
every run and never records a deployment (its only metadata footprint is
the overlay manifest it uses to clean up after itself). `test` and
`explain` never touch a remote at all — they run against a local Docker
container.

### Crate layering

```
bin/mz-deploy/main.rs        clap dispatch, Settings loading
        │
cli/commands/*               one module per command
        │
cli/executor.rs              shared plan-then-execute machinery for apply commands
        │
   ┌────┴──────────────┐
project/*                client/*
compile pipeline:        connection + domain sub-clients:
syntax → resolve →       deployments, introspection,
compiler → ir →          provisioning, validation,
analysis                 type_info, dev_overlays
   └────┬──────────────┘
config.rs  fs.rs  log.rs  diagnostics.rs  secret_resolver/  docker_runtime.rs
```

`project` knows nothing about connections; `client` knows nothing about SQL
files on disk. Commands are the only place the two meet: a typical deploy
command compiles the project, asks `client` what exists on the server,
diffs, and issues DDL.

### Configuration and profiles

Two TOML files configure everything (`src/config.rs`):

- **`project.toml`** (project root, checked in): the Materialize version
  (`mz_version`, which selects the Docker image for `test`/`explain`),
  declared external `[dependencies]` (objects the project references but
  does not define), and per-profile sections carrying `variables`
  (substituted into SQL), an optional `profile_suffix` (rewrites database
  and cluster names, enabling several copies of one project in a single
  environment), security config (`aws_profile` for secret resolution), and
  an `emulator` flag that relaxes RBAC checks.
- **`profiles.toml`** (`~/.mz` or `--profiles-dir`, machine-local): named
  connection profiles — host, port, user, password (supports `${VAR}` env
  expansion), TLS settings, and an optional `http_host` used only by `mcp`.

The active profile resolves in order: `--profile` flag → `MZ_DEPLOY_PROFILE`
env var → a project-local `.mzprofile` file (git-ignored, written by
`mz-deploy profile set`). A built-in `emulator` profile points at
`localhost:6875` so local development needs no configuration.

### The connection layer

`client/connection.rs` wraps a single `tokio_postgres` connection in a
`Client` and exposes domain-grouped borrowed sub-clients
(`client.deployments()`, `.introspection()`, `.provisioning()`,
`.validation()`, `.types()`, `.dev_overlays()`). Every connection pins its
session to the `_mz_deploy_server` cluster via libpq options so metadata
queries never land on (or wake) user compute; `connect_with_profile_no_pin`
exists for the two cases where that cluster does not exist yet (`setup`,
and the ephemeral Docker container used by `test`/`explain`). TLS mode
defaults to prefer for localhost and require for remote hosts, with CA
discovery falling back from `sslrootcert` to platform stores to OpenSSL
defaults.

### Roles and authorization

`setup` creates three roles on RBAC-enabled environments, and commands gate
on them (`cli/commands/setup.rs::require_deployer` / `require_developer`):

| Requirement | Commands |
|---|---|
| superuser | `setup` |
| `materialize_deployer` | `stage`, `promote`, `abort`, `delete`, and the entire `apply` family (enforced in `cli/executor.rs` before any phase runs) |
| `materialize_developer` + `CREATEDB` | `dev` |
| membership in any mz-deploy role | read-only commands (`list`, `describe`, `log`, `wait`), via the grants `setup` installs on `_mz_deploy`; `materialize_monitor` exists for exactly this |

Role detection (`setup::validate_connection`) requires the caller to be a
member of *exactly one* of the three roles — membership in several is an
error, and superusers bypass the check. Profiles with `emulator = true`
skip role enforcement entirely — intended for the single-user local
emulator, where the roles don't exist.

### Server-side state

All deployment state lives in a `_mz_deploy` database on the target server,
created by `setup` (`cli/commands/setup_schema.rs`):

| Table (`_mz_deploy.tables.*`) | Purpose |
|---|---|
| `deployments` | One row per staged deployment per schema: deploy id, who, when, git commit, kind, `promoted_at` (NULL while staging) |
| `objects` | Append-only log of `(deploy_id, object, semantic hash)` — the production snapshot that changesets diff against |
| `clusters` | Cluster ids belonging to each deployment |
| `pending_statements` | Deferred DDL (sinks) to execute after the swap, with `executed_at` for idempotent resume |
| `replacement_mvs` | Materialized views to update in place via `APPLY REPLACEMENT` after the swap |
| `dev_overlays` | Manifest of per-developer overlay databases so `dev` can always find and drop its previous run |
| `version` | Metadata schema version |

Views in `_mz_deploy.public.*` derive the interesting projections:
`production` (latest promoted deployment per schema), `staging_deployments`
(`promoted_at IS NULL`), `deployment_clusters`, and `missing_clusters`,
which left-joins recorded cluster ids against `mz_catalog.mz_clusters` to
expose deployments whose clusters were dropped out-of-band. All reads and
writes go through `client/deployment_ops.rs`.

The `kind` column distinguishes per-schema deployment rows: `stage` writes
`Objects` (regular views/MVs), `Sinks` (schemas whose only changes are
deferred sinks — nothing to swap), and `Replacement`. A
*replacement-managed* (or *stable*) schema is one declared
`ALTER SCHEMA ... SET api = stable` in its mod file; its materialized views
are updated in place rather than swapped — full treatment in
[stable schemas](#stable-schemas-and-apply-replacement) below. A `Tables`
variant exists in `client/models.rs` but no command currently records it.

Because the metadata lives in the same server as the objects it describes,
conflict detection and crash recovery are plain SQL — no external lock
service, no local state besides the project directory. There is also no
process-level locking: nothing stops two mz-deploy invocations from running
at once. Three mechanisms make that safe. Concurrent stages of *different*
deploy ids own disjoint suffixed resources; a stage of an *existing*
unpromoted id is rejected up front; and concurrent promotes that touch the
same production schemas surface through promote-time conflict detection
(below). The window that is not defended is two promotes of different
deployments over disjoint schemas racing each other's marker bookkeeping —
each owns its own `apply_<id>_*` schemas, so even that interleaves safely
at the metadata level.

The `tables.version` row records the metadata schema version (seeded to 1
by `setup` with a pre-check). There is no migration machinery yet: a newer
mz-deploy meeting an older metadata schema relies on rerunning `setup`,
whose `IF NOT EXISTS`-guarded statements create whatever is missing.

When server reality drifts from the metadata — a staging cluster deleted by
hand — `validate_deployment_clusters` (`client/deployment_ops.rs`) catches
it via the `missing_clusters` view and fails with instructions to `abort`
the deployment; staged *schemas* dropped by hand are tolerated, since
`abort`'s drops are `IF EXISTS`-style and idempotent.

### The compilation pipeline

`mz-deploy compile` turns a project directory into a validated,
dependency-ordered graph of objects — without contacting a Materialize
server. Every other command builds on this: `stage` compiles before it
diffs, `test` compiles before it lowers, the LSP recompiles on every idle
pause. The pipeline is incremental end to end, so the cost of a run is
proportional to what changed, not to project size.

```
cli/commands/compile.rs::run()
  └─ project::plan_sync()
      └─ project/compiler.rs::compile_sync_with_stats()
          1. discover_project()       walk models/, collect ObjectDescriptors
          2. fingerprint (parallel)   match each descriptor against the cache
          3. load cache hits          re-parse cached SQL; demote failures to misses
          4. compile misses (parallel) parse → normalize → validate → persist
          5. assemble_project()       combine into compiled::Project
          6. suffix rewrites          database + cluster name maps (profile_suffix)
          7. graph::Project::from()   extract dependencies, build the DAG
  └─ typecheck::run()                 incremental, offline; skipped by apply
```

The compiler produces two project representations in sequence:
`ir/compiled.rs::Project` (validated objects grouped by database/schema, no
edges yet) and `ir/graph.rs::Project` (the final output: hierarchical
databases → schemas → objects plus a flat
`dependency_graph: BTreeMap<ObjectId, BTreeSet<ObjectId>>`, external
dependencies, cluster dependencies, and collected unit tests).

#### Discovery

`discover_project()` (`project/compiler.rs`) walks `models/`. Directory
structure determines identity: `models/<db>/<schema>/<object>.sql` declares
`db.schema.object`, and `ir/compiled.rs::FullyQualifiedName` later verifies
that the CREATE statement inside actually names that object. Two kinds of
companion files participate:

- `<db>/<db>.sql` and `<schema>/<schema>.sql` hold database- and
  schema-level statements (for example `ALTER SCHEMA ... SET api = stable`,
  which marks a schema's materialized views as replacement-managed). These
  are parsed and validated on every run, never cached; they are small, and
  skipping the cache avoids staleness.
- `<object>#<profile>.sql` is a per-profile variant of an object. All
  variants of one object are gathered into a single `ObjectDescriptor`
  (object key + list of `(path, Option<profile>)` variants), which is the
  unit of both parallelism and caching.

If the active profile sets `profile_suffix`, discovery records a
database-name map (`app` → `app_<suffix>`) that later stages use to rewrite
cross-database references, so one environment can host several independent
copies of a project.

#### Incremental caching

The cache (`project/compiler/cache/build_artifact.rs`) is a SQLite database
under `target/compiler/` in the project root, namespaced per (profile,
suffix, variables) tuple, so switching profiles never poisons another
profile's artifacts. Four tables:

| Table | Holds |
|-------|-------|
| `file_state` | path, size, mtime, content hash, contents |
| `object_state` | the compiled artifact: primary CREATE statement SQL plus fragment lists (indexes, grants, comments, tests) |
| `typecheck_objects` | per-object typecheck fingerprints |
| `typecheck_columns` | per-object output column schemas |

An object's fingerprint covers its key, every variant's path and content
hash, the compile-time variable map, and the active profile/suffix (the
last two repeat the namespace dimensions inside each entry — deliberate
defense in depth). Paths being part of the fingerprint means relocating a
checkout invalidates the cache — also deliberate, since absolute paths can
affect behavior. Matching is two-phase: all descriptors fingerprint in
parallel (rayon), then full artifacts are loaded only for hits, and a hit
whose cached SQL fails to re-parse is demoted to a miss rather than
erroring. Artifacts store SQL strings, not serialized ASTs, so cache
compatibility survives AST type changes.

`cache/project_cache.rs` is the read-only face of the same database, used
by the LSP, `test`, and `explain` to look up compiled objects and column
schemas without recompiling.

#### Parsing and variables

Before parsing, `project/syntax/variables.rs` substitutes psql-style
variables — `:name` (raw), `:'name'` (string literal), `:"name"`
(identifier) — from the active profile's `[<profile>.variables]` table,
skipping occurrences inside quotes, comments, and dollar-quoted strings.
The resolver tracks every substitution's offset mapping so diagnostics in
resolved SQL can point at the original source, and records unresolved
variables (an error unless the file opts into
`PRAGMA WARN_ON_MISSING_VARIABLES`).

Parsing itself is `mz_sql_parser` — the real Materialize parser, not an
approximation — producing `LocatedStatement`s (AST + byte offset). The AST
layer (`project/ast.rs`) wraps the statement variants the project model
accepts; `EXECUTE UNIT TEST` statements are extracted here and carried on
the owning object (see [unit testing](#unit-testing)).

#### Normalization and validation

`project/resolve/normalize.rs` provides `NormalizingVisitor`, an AST
visitor with several strategies used across the codebase: *fully
qualifying* (rewrite every reference to `database.schema.object`, the form
stored in artifacts), *flattening* (rewrite to a single quoted identifier
`"db.schema.object"`, used by test lowering), and explain/staging variants.
`resolve/cte_scope.rs` tracks CTE bindings so a reference to a local CTE is
never mistaken for a dependency.

Per-object validation (`project/compiler/object_validation.rs`) checks
identifier legality, statement shape, and that the declared name matches
the file path; errors accumulate across all objects and report together
rather than failing fast. `assemble_project()` then merges objects with
database/schema metadata into `compiled::Project`, after which the
profile-suffix database and cluster maps are applied via
`rewrite_database_references` / `rewrite_cluster_references`.

#### Dependency analysis

`project/analysis/deps.rs` converts `compiled::Project` into
`graph::Project`. Dependency extraction walks each statement with the
`mz_sql_parser` visit trait, collecting table references while the CTE
scope excludes local names. References to objects not defined in the
project become *external dependencies* and are cross-checked against
`[dependencies]` in `project.toml`: undeclared externals are errors,
declared-but-unused ones are warnings.

Three validations matter for deployment correctness:

- `analysis/topology.rs` topologically sorts the graph (dependencies before
  dependents) and rejects cycles.
- `analysis/graph_validation.rs::validate_cluster_isolation()` forbids
  sources/sinks from sharing a cluster with views/MVs/indexes. The
  blue/green swap operates on whole clusters, and mixing storage and
  compute objects on one cluster would force source recreation during an
  object-only deploy.
- `compiler/object_validation/schema_constraints.rs` forbids *storage*
  objects (tables, sinks) from sharing a schema with *computation* objects
  (views, materialized views). This is the invariant that makes the
  deployment lifecycle's `DROP SCHEMA ... CASCADE` safe: a schema that
  participates in a blue/green swap can only ever contain recomputable
  objects, so dropping the old copy can never destroy data.

#### Snapshots

Deployment commands need to know *what changed* relative to production.
`analysis/deployment_snapshot.rs` defines the comparison basis: a map from
`ObjectId` to a semantic content hash computed from the parsed AST, so
whitespace and comment edits do not register as changes. Tables, sources,
secrets, and connections are excluded — they are apply-managed, not staged.
Snapshots of what production currently runs are loaded from
`_mz_deploy.tables.{deployments,objects}` on the server. The changeset
computation that diffs two snapshots belongs to the deployment commands,
not to `compile` — it runs inside `stage` and `dev`, and is described in
[changesets](#changesets-deciding-what-to-deploy) below.

#### Error model

`project/error.rs` defines the hierarchy (`Load`, `Parse`, `Validation`,
`Dependency`), and `diagnostics.rs` localizes each error kind to a byte
range in the original file — using the parser's offset bookkeeping and,
when necessary, whole-word identifier search — then renders it with
`annotate_snippets`, including "did you mean" suggestions built from the
known-identifier universe.

### Offline typechecking

`project/compiler/typecheck.rs` typechecks SQL without a server by running
Materialize's real planner against an in-memory stub catalog
(`typecheck/catalog.rs`, the largest module in the crate). The actual
typecheck of one object is `catalog.rs::resolve_plan_and_insert()`: it
name-resolves the statement's AST with `mz_sql::names::resolve`, then calls

```rust
mz_sql::plan::plan(Some(&pcx), &catalog, resolved, &Params::empty(), &resolved_ids)
```

— the same planning entry point `environmentd` uses — and extracts the
resulting `RelationDesc` as the object's output columns. Anything the
planner would reject on a real server (unknown columns, type errors,
unsupported constructs) fails here, before deployment.

The stub catalog implements just enough of the `mz_sql` catalog traits for
planning: builtin functions and types come from
`mz_catalog::builtin::BUILTINS`, and every object a statement depends on is
present as a stub base relation with known columns. Column types for
*external* dependencies come from `types.lock` — a checked-in file
generated by `mz-deploy lock`, which queries a live server's
`mz_catalog.mz_columns` (`client/type_info.rs`) for the declared
dependencies.

#### Incremental typechecking

`typecheck::run()` only re-plans what could have changed. It computes an
**initial dirty set**: a view/MV is dirty if (1) its own source changed
this compile (`project.compile_dirty`), (2) it has no cached typecheck
row — never validated, or its last attempt failed — or (3) a *non-view*
direct dependency was recompiled or an external dependency's schema
changed. External changes are detected by diffing SHA-256 digests of each
`types.lock` entry's column map against digests cached from the previous
run, so editing the lock file dirties exactly the objects that read the
changed tables. Dirty *view* dependencies are deliberately not part of the
initial set — they are reached by the pessimistic expansion below and then
filtered by schema-stability gating, which is more precise.

The initial set then expands to a **pessimistic dirty set**: every
transitive view dependent, walked over the reverse dependency graph, since
an upstream schema change *may* cascade. The pessimism is corrected at
execution time by **schema-stability gating**: when a dirty node
re-typechecks, its freshly planned columns are compared against its cached
columns. If they match, the node is marked `schema_stable`, and a dependent
re-typechecks only if at least one of its dependencies was *not*
schema-stable. Editing a leaf view's filter therefore re-plans that one
view; the rest of the project skips re-planning and returns cached columns.
`TypecheckStats` (`ran`/`skipped` and, within ran,
`schema_stable`/`schema_changed`) reports exactly how much the gating
saved.

#### Parallel execution

The dirty work runs on a ready-queue DAG executor
(`typecheck/executor.rs`). Bootstrap is serial:
`bootstrap::bootstrap_catalog()` builds one shared base catalog seeding
builtins, namespaces, external types, and stubs for every non-view
dependency of the dirty set. Then each view/MV in the DAG is a node holding
a `OnceLock` result slot and an atomic remaining-dependencies counter.
Nodes with zero remaining deps go on a crossbeam channel; a rayon scope
spawns one worker per thread, and each worker pulls a node, runs the
typecheck closure with its direct dependencies' results, stores the
outcome, and decrements its dependents' counters — enqueuing any that
reach zero. Parallelism is therefore bounded by the dependency graph's
width, not by file count.

Per node, `typecheck_node()` builds a `TaskCatalog` — a cheap writable
layer over the shared `Arc<CatalogRuntime>` base — and stubs each direct
dependency as a table using the *dependency's freshly computed columns*
from the DAG, so a node always plans against its upstreams' current
schemas, never stale cache. Each node's catalog layer is private, so no
state leaks between objects. A node whose own validation fails produces
`Failed`; its downstreams produce `Blocked` rather than spurious cascading
errors, and both keep their last successful cached row. Only nodes whose
schema actually changed (or are new) are written back to SQLite.

One trust boundary to keep in mind: the typechecker believes `types.lock`.
If an external dependency's columns change on the server after the lock
was generated, compilation passes against the stale schema and the
mismatch only surfaces when the object is actually created during deploy.
Rerunning `mz-deploy lock` (or a full `apply`, which refreshes it) is the
remedy.

`apply` commands compile with typechecking disabled
(`compile::run_without_typecheck`): the infrastructure they create is what
later compilations typecheck against, so checking before it exists would
be circular.

### Changesets: deciding what to deploy

Both `stage` and `dev` start by asking what changed relative to production.
The answer comes from `analysis/changeset.rs`:
`ChangeSet::from_deployment_snapshot_comparison()` diffs the production
snapshot (object → AST-level semantic hash, loaded from
`_mz_deploy.tables.{deployments,objects}`) against the freshly compiled
project, then runs a Datalog fixed-point over the project's dependency
graph to compute the transitive blast radius. Three relations are derived
(`changeset/datalog.rs`, base facts from `changeset/base_facts.rs`):

(Read `Head :- Body1, Body2` as "Head holds whenever every Body fact
holds"; the rules iterate to a fixed point.)

Statement dirtiness — which objects must be redeployed:

```datalog
DirtyStmt(O) :- ChangedStmt(O)                              % changed objects are dirty
DirtyStmt(O) :- StmtUsesCluster(O, C), DirtyCluster(C)      % objects whose statement runs on a dirty cluster
DirtyStmt(O) :- DependsOn(O, P), DirtyStmt(P),
                NOT IsChangedReplacement(P)                 % dependents, except through changed replacement MVs
DirtyStmt(O) :- DirtySchema(Db, Sch), ObjectInSchema(O, Db, Sch),
                NOT IsReplacement(O)                        % whole schemas redeploy atomically
```

Cluster dirtiness — which clusters get a staged twin (`I` ranges over an
object's indexes):

```datalog
DirtyCluster(C) :- ChangedStmt(O), StmtUsesCluster(O, C),  NOT IsSink(O), ClusterBoundary(C)
DirtyCluster(C) :- ChangedStmt(O), IndexUsesCluster(O, I, C), NOT IsSink(O), ClusterBoundary(C)
```

Schema dirtiness — which schemas participate in the swap:

```datalog
DirtySchema(Db, Sch) :- DirtyStmt(O), ObjectInSchema(O, Db, Sch), NOT IsSink(O), NOT IsReplacement(O)
```

The negations carry the design intent:

- **`StmtUsesCluster`, not `IndexUsesCluster`, feeds rule 2 of statement
  dirtiness.** An object whose *index* sits on a dirty cluster is not
  itself dirty — indexes are physical optimizations, and moving one must
  not redeploy the indexed object.
- **Clusters only become dirty from `ChangedStmt`, never from derived
  `DirtyStmt`.** An object that is dirty merely because its schema is dirty
  does not drag its cluster into the deployment. `ClusterBoundary`
  restricts this further to clusters actually referenced by the project.
- **`NOT IsSink` appears in both cluster and schema rules.** Sinks are
  created after the swap, so a sink edit must not dirty its schema (which
  would force a blue/green redeploy of everything in it) or its cluster.
- **`NOT IsChangedReplacement(P)` stops propagation at stable-API
  boundaries.** A changed materialized view in a replacement-managed schema
  is updated in place (see below), so its dependents do not redeploy.
- **`NOT IsReplacement(O)` keeps replacement MVs out of the swap
  machinery.** A replacement MV neither dirties its schema nor becomes
  dirty just because its schema is — it is deployed by its own protocol
  (`APPLY REPLACEMENT`), never by schema swap.

Before the fixed-point runs, the changed-replacement set is computed with a
guard: an object counts as a *changed replacement* only if it existed in
the old snapshot **and** its schema was already `Replacement` kind there.
An object newly added to a stable schema, or a schema transitioning into
`api = stable`, goes through the normal blue/green swap — `APPLY
REPLACEMENT` needs an existing production target to replace. The resulting
`ChangeSet` carries that distinction as `new_replacement_objects` (swap)
vs `changed_replacement_objects` (in-place replacement).

### The deployment lifecycle

mz-deploy deploys views and materialized views blue/green. `stage` builds
the new world in suffixed schemas and clusters alongside production; `wait`
watches the new clusters hydrate; `promote` swaps staging and production
atomically and then performs the work that could not happen before cutover.
Production is never modified until the single swap transaction commits, and
every step after it is resumable.

The design depends on four server-side primitives. If they break, the tool
breaks. `ALTER SCHEMA/CLUSTER ... SWAP WITH` exchanges two objects' names
and must compose transactionally — several swaps in one `BEGIN`/`COMMIT`
are all-or-nothing, and this is the entire atomicity argument.
`ALTER MATERIALIZED VIEW ... APPLY REPLACEMENT` replaces an MV's
computation while preserving its identity, so dependents are unaffected.
`ALTER SINK ... SET FROM` atomically switches a running sink's input
without disrupting its connection to the external system; if that switch
is not clean, surviving sinks emit garbage during promotion. And
`COMMENT ON SCHEMA` rides along in the same transactions, which is what
makes the crash-recovery marker below sound. Reviewing this crate means
also being satisfied those server behaviors hold.

#### stage (`cli/commands/stage.rs`)

`stage` deploys the *changed* subset of the project into parallel
infrastructure. The flow:

1. **Pre-flight.** Refuse a dirty git tree unless `--allow-dirty`. The
   deploy id defaults to the 7-char git SHA (random hex without git), and
   everything staged carries the suffix `_<deploy_id>`. Compile the
   project, connect, verify `setup` has run, and validate the caller's
   role.
2. **Changeset.** Load the production snapshot and compute the dirty sets
   as described above. An empty changeset exits early as a no-op — staging
   when production already matches the project is free. (Staging a commit
   that is already *staged but unpromoted* is different: that hits the
   duplicate-id rejection below.)
3. **Partition.** Changed objects split four ways (`partition_objects()`):
   tables/sources/secrets/connections are *skipped* (apply-managed); sinks
   are *deferred* — a sink created in staging would immediately write to
   its external system, so its CREATE statement is recorded in
   `pending_statements` instead; materialized views in replacement-managed
   schemas (`ALTER SCHEMA ... SET api = stable`) are recorded in
   `replacement_mvs`; everything else (views, MVs) is staged.
4. **Validate.** `client/validation.rs` checks ownership and privileges on
   the affected schemas and clusters, cluster isolation, and that sink
   connections exist — before any DDL is issued.
5. **Record, then create.** Metadata rows (deployment, objects, clusters,
   pending statements, replacement MVs) are inserted *before* resources are
   created, so a partially failed stage is always visible to `abort`. Then
   staging schemas (`<schema>_<deploy_id>`) and staging clusters (cloned
   from the production cluster's size, replication factor, and grants) are
   created, and objects are deployed in topological order. Any failure
   triggers automatic rollback of created resources unless `--no-rollback`.

Concurrent staging is safe by construction: each deploy id owns disjoint
suffixed resources, and conflicts are deferred to promote time. The one collision
case — two stages of the same deploy id, e.g. two engineers staging the
same commit — is rejected up front: `stage` fails with "deployment already
exists" if an unpromoted deployment with that id is already recorded.

#### wait (`cli/commands/wait.rs`)

`wait` blocks until a deployment's staging clusters are ready to serve.
Rather than polling, it opens a `SUBSCRIBE` over a query against
`mz_internal.mz_hydration_statuses` joined with replica status
(`client/introspection.rs::subscribe_deployment_hydration`), streaming
per-cluster transitions into a live dashboard. A cluster is:

- **Hydrating** — some objects not yet hydrated (`hydrated/total` shown);
- **Lagging** — hydrated but wallclock lag exceeds `--allowed-lag`;
- **Ready** — hydrated and within lag;
- **Failing** — no replicas, or every replica is OOM-looping (3+ kills in
  24h per `mz_cluster_replica_status_history`).

`--once` takes a single snapshot instead of streaming; `--timeout` bounds
the wait; `--output json` emits one JSON object per update. Exit code 0
means every cluster is Ready. Promotion reuses this as a hard gate:
`main.rs` runs `wait::run(..., once=true, ...)` before calling
`promote::run`, so a cluster that is Hydrating, Lagging beyond
`--allowed-lag` (default 300s), or Failing aborts the promote unless
`--no-ready-check` is passed.

#### promote (`cli/commands/promote.rs`)

Promotion is one atomic transaction followed by resumable cleanup work. The
hard problem is that some of the work (creating sinks, updating replacement
MVs, dropping the old world) cannot be inside the swap transaction, so the
command must survive being killed at any point and resume correctly.

**Conflict detection.** Before anything else within `promote::run` (the
readiness gate above has already run), promote checks whether any
production schema this deployment touches was promoted by *someone else
after this staging began* (`check_deployment_conflicts`). That is
last-writer-wins detection with git-merge ergonomics: the conflict lists
who promoted what and when, and `--force` overrides it knowingly.

**Dummy schemas as a progress record.** Promote needs durable,
transactional state to record how far it got. It uses the catalog itself:
two *empty* schemas created in the `_mz_deploy` database,
`apply_<deploy_id>_pre` (comment `swapped=false`) and
`apply_<deploy_id>_post` (comment `swapped=true`).
(The `apply_` prefix is historical and unrelated to the `mz-deploy apply`
command.) The schemas hold no objects — their existence and their comments
*are* the state. Because schema creation, schema swap, and comments are all
ordinary catalog operations, the progress record participates in the same
transaction as the real work. The swap transaction includes
`ALTER SCHEMA _mz_deploy.apply_<id>_pre SWAP WITH apply_<id>_post`, which
exchanges the two schemas' names and therefore moves the `swapped=true`
comment onto `_pre` atomically with the production cutover. There is no
instant at which production is swapped but the marker says otherwise, or
vice versa. On startup, `get_apply_state()` (`client/deployment_ops.rs`)
reads the marker: `NotStarted` (no marker schemas), `PreSwap` (`_pre` says
`swapped=false` — the swap has not committed, redo it), or `PostSwap`
(`_pre` says `swapped=true` — the swap committed, skip straight to
post-swap work). The markers are dropped only at the very end, after all
post-swap work succeeds.

**The swap transaction.** In a single `BEGIN`/`COMMIT`:

```sql
ALTER SCHEMA "<db>"."<schema>" SWAP WITH "<schema>_<deploy_id>";   -- per schema
ALTER CLUSTER "<cluster>" SWAP WITH "<cluster>_<deploy_id>";       -- per cluster
ALTER SCHEMA _mz_deploy."apply_<id>_pre" SWAP WITH "apply_<id>_post";
```

Production names now point at the staged objects; the old objects live on
under the staging names. Deployments of kind `Sinks` and steady-state
`Replacement` schemas are excluded from the swap — they have nothing to
swap.

**Post-swap work** (`run_post_swap_steps()`), sequential and idempotent.
Each step's resume safety has a specific mechanism, worth checking
individually in review:

1. Execute `pending_statements` — the deferred `CREATE SINK`s, now reading
   from promoted objects (idempotency layers in
   [the sink lifecycle](#how-sinks-move-through-a-deployment) below).
2. For each `replacement_mvs` row, run
   `ALTER MATERIALIZED VIEW <target> APPLY REPLACEMENT <staged copy>` —
   the MV's computation is replaced in place while its identity (and
   therefore everything downstream of it) is preserved — then drop the
   now-empty replacement staging schemas. (See
   [stable schemas](#stable-schemas-and-apply-replacement) below.)
3. Repoint sinks that read from objects in the old, about-to-be-dropped
   schemas: `repoint_dependent_sinks()` re-queries the catalog post-swap
   for sinks depending on the old schemas, verifies the promoted
   counterpart of each input exists, and issues
   `ALTER SINK ... SET FROM <promoted object>`. The sink itself is never
   dropped or recreated — it atomically switches inputs and keeps writing
   to the same external resource. If a promoted counterpart is missing —
   the sink read an object this deployment removed — promote fails rather
   than leaving the sink pointed at a schema it is about to drop; the
   operator unblocks by dropping the sink or restoring the object, then
   rerunning `promote <id>`.
4. Set `promoted_at` on the deployment rows.
5. Drop the old world: `DROP SCHEMA <schema>_<deploy_id> CASCADE` and the
   old clusters. The `CASCADE` is safe because swapped schemas can only
   ever contain computation objects — the compiler's storage/computation
   isolation rule guarantees no table or sink lives in a schema that
   participates in the swap, so dropping the old copies destroys only
   recomputable views and MVs.
6. Drop the marker schemas and delete the deferred-work rows.

If any step fails, rerunning `mz-deploy promote <id>` resumes from the
marker state and retries the remaining steps.

#### Stable schemas and APPLY REPLACEMENT

Blue/green has a scaling problem at project boundaries. Within one
project, a changed object's dependents are detected as dirty and
redeployed with it, so nothing breaks. But the cascade grows with the
project. And consumers in *other* projects break outright: this project
cannot see them, let alone redeploy them.

`ALTER SCHEMA ... SET api = stable` (declared in the schema's mod file,
`models/<db>/<schema>/<schema>.sql`) marks a schema as a stable API
boundary that other teams may depend on. The compiler enforces the
contract's precondition: a stable schema may contain *only* materialized
views — no tables, views, sinks, or sources — because only an MV has a
durable identity whose computation can be replaced underneath it.
(`SET api` itself is an mz-deploy directive; `analysis/topology.rs` strips
it from what is sent to the server.)

For a *changed* MV in a stable schema, the lifecycle changes shape:

- **stage** still builds the new definition in a staging schema and records
  a `replacement_mvs` row mapping the production target to its staged copy,
  with the deployment row marked kind `Replacement`.
- **promote** excludes steady-state `Replacement` schemas from the swap.
  After the swap, `apply_replacement_mvs()` runs, per row:

  ```sql
  ALTER MATERIALIZED VIEW "<db>"."<schema>"."<mv>"
    APPLY REPLACEMENT "<db>"."<schema>_<deploy_id>"."<mv>";
  ```

  The server replaces the production MV's computation with the staged
  copy's while preserving the MV's identity — downstream consumers, in this
  project or any other, are untouched and never know an update happened.
  The emptied replacement staging schemas are then dropped.

Two changeset rules back this up. `IsChangedReplacement` blocks dirtiness
propagation, so a changed stable MV does not redeploy its dependents. The
new-vs-changed guard decides which MVs take this path at all, since
`APPLY REPLACEMENT` needs an existing production target.

#### How sinks move through a deployment

Sinks write to external systems, which makes them the one object kind that
cannot be blue/greened: a staged sink would start producing immediately,
emitting data derived from unpromoted objects to a real downstream
consumer. Every special case in the lifecycle follows from that fact.

- **Changeset:** sinks are excluded from both cluster and schema dirtiness
  (the `NOT IsSink` negations above). Editing a sink redeploys the sink —
  nothing else.
- **stage:** `partition_objects()` diverts sinks out of the staged set.
  Their `CREATE SINK` statements are recorded in
  `_mz_deploy.tables.pending_statements` with a sequence number and
  `executed_at = NULL`; a schema whose only changes are sinks gets a
  deployment row of kind `Sinks` and no staged twin at all.
- **promote, after the swap:** pending statements run with two layers of
  idempotency — only `executed_at IS NULL` rows are fetched, and sinks that
  already exist in the catalog are skipped — each marked via
  `mark_statement_executed` after it runs. By this point the swap has
  committed, so a new sink's first read is of promoted objects.
- **promote, for surviving sinks:** an *unchanged* sink keeps running
  through the swap — it references its input by identity, so after the swap
  it is reading from the old object now living under the staging name.
  Before dropping the old schemas, promote repoints each such sink with
  `ALTER SINK ... SET FROM`, atomically switching it to the promoted object
  while it keeps its own identity and its connection to the external
  resource.

The net effect: a sink is created exactly once, against promoted data, and
an existing sink experiences a deployment as a single atomic input switch
rather than a drop-and-recreate.

#### abort (`cli/commands/abort.rs`)

`abort` discards an unpromoted staging deployment: it validates the
deployment exists and is not promoted, drops the staging schemas (CASCADE)
and clusters, and deletes every metadata row for the deploy id. All drops
are `IF EXISTS`-tolerant, so abort is safe to rerun over a partially
rolled-back stage.

#### dev (`cli/commands/dev.rs`)

`dev` is the scratch loop for working against a remote environment without
the stage/promote ceremony. Each run:

1. Requires the developer role and refuses to target any cluster currently
   hosting promoted deployments (`list_production_clusters`) — overlays
   must land on scratch compute.
2. Computes the dirty set the same way `stage` does, filtered to views and
   MVs only.
3. Drops the previous overlay databases recorded in
   `_mz_deploy.tables.dev_overlays` for this (profile, project) pair —
   drop-and-rebuild every run, no incremental overlay state to corrupt.
4. Creates `<database>__<profile>` overlay databases containing only the
   dirty objects, with references rewritten (`client/dev_overlays.rs`):
   dirty in-project references point into the overlay, clean ones resolve
   to real production objects, and every `IN CLUSTER` clause is rewritten
   to the requested scratch cluster.

The result: a personal copy of what you changed, layered over live
production data, rebuilt in seconds. `dev --down` drops the overlay.

#### Failure-mode summary

| Interrupted at | Detected by | Recovery |
|---|---|---|
| stage fails mid-creation | rollback drops created resources *and deletes the metadata rows* (`rollback_staging_resources`) | rerun `stage` |
| stage fails with `--no-rollback`, or rollback itself dies | metadata and partial resources persist | `abort <id>`, then rerun `stage` |
| promote before swap committed | marker comment `swapped=false` | rerun `promote <id>` — swap retried |
| promote after swap, before cleanup | marker comment `swapped=true` | rerun `promote <id>` — post-swap steps retried |
| deferred sink/replacement failure | `executed_at IS NULL` rows remain | rerun `promote <id>` |
| old-world drop failure | old schemas linger under staging names | rerun `promote <id>` or drop manually |
| dev crash | `dev_overlays` manifest rows remain | next `dev` run drops them |

#### What can destroy data

For reviewers auditing destructive paths, the complete list of drops:

- **promote** drops the old production schemas `CASCADE` and old clusters —
  safe by the storage/computation isolation invariant above (only views/MVs
  can be in them), but it *will* take any ad-hoc object someone manually
  created in a managed schema.
- **abort** drops a deployment's staging schemas `CASCADE` and staging
  clusters — by construction these contain only this deployment's staged
  copies.
- **delete** drops a named infrastructure object (cluster, connection, ...)
  after confirmation; this is the only path that removes apply-managed
  objects, and tables/sources go with their data.
- **dev** drops and rebuilds its overlay databases on every run; `dev
  --down` drops them without rebuilding. Overlays never contain tables.
- **clean** is local-only (the `target/` compiler cache).

`apply` never drops anything; `stage` drops only what it created in the
same run (rollback).

### Infrastructure management: apply, setup, and secrets

Where `stage`/`promote` manage the project's *logic* (views and
materialized views) blue/green, the `apply` family manages its
*infrastructure* — the long-lived, stateful objects that cannot be swapped:
clusters, roles, network policies, secrets, connections, sources, and
tables. Apply is idempotent reconciliation: it creates what is missing,
alters what has drifted, and never drops anything (deletion is the explicit
`delete` command).

#### Orchestration

`cli/commands/apply_all.rs` runs every phase in dependency order:

```
clusters → roles → network policies → secrets → connections → sources → tables
```

Where each phase's definitions live on disk: clusters, roles, and network
policies have their own top-level directories (`clusters/`, `roles/`,
`network-policies/`); secrets, connections, sources, and tables are
ordinary objects in `models/` — they are apply-managed because of *what*
they are, not where they live. (See the
[statement-taxonomy table](#the-project-model) for what each file kind
accepts.)

Each phase is also independently invocable (`mz-deploy apply clusters`,
etc.). The shared machinery is `cli/executor.rs`: a command builds an
`ApplyPlan` by compiling the project (with typechecking disabled — the
objects being created are what later compiles typecheck against), preparing
required databases and schemas via `client/provisioning.rs`, and adding
phases of planned statements. The plan can be displayed without execution
(`--dry-run`) or executed, producing per-object results
(`Created`/`Altered`/`UpToDate`/`Skipped`) with the statements that ran —
secrets carry parallel redacted statements so values never appear in output
or JSON. Objects sharing a `transaction_group` execute inside one
`BEGIN`/`COMMIT`.

The general per-object pattern in every phase is *introspect, diff, emit*:
check existence via `client/introspection.rs`, compare desired
configuration against live state, and emit `CREATE`, `ALTER`, or nothing.
Grants and comments are reconciled on every phase through the shared
`cli/commands/grants.rs` logic, which computes `GRANT`/`REVOKE` deltas
rather than reissuing everything.

#### Per-phase behavior

- **Clusters** (`cli/commands/clusters.rs`): definitions come from the
  `clusters/` directory. Existing clusters are diffed on size and
  replication factor and reconciled with `ALTER CLUSTER ... SET`; grants
  and comments follow.
- **Roles** (`cli/commands/roles.rs`) and **network policies**
  (`apply_network_policies.rs`): same create-or-reconcile shape.
- **Secrets** (`apply_secrets.rs`): values are resolved client-side at
  apply time (below), then `CREATE SECRET ... AS <literal>` for new secrets
  or `ALTER SECRET ... SET VALUE` for existing ones. There is no drift
  detection — Materialize cannot reveal a secret's current value — so apply
  always rewrites it.
- **Connections** (`apply_connections.rs`): existing connections are diffed
  option-by-option against `SHOW CREATE CONNECTION` output and reconciled
  with `ALTER CONNECTION ... SET/DROP`.
- **Sources** (`apply_sources.rs`) and **tables** (`apply_tables.rs`):
  create-only — a source or table that exists is left alone, since altering
  one is generally a data-loss decision a tool should not make silently.
  `CREATE TABLE ... FROM SOURCE` statements are grouped by source so they
  apply in a sensible order, and indexes/grants/comments follow creation.

After a successful full apply, `lock::run()` refreshes `types.lock` so the
offline typechecker knows the column types of the objects that now exist.

#### Secret resolution (`secret_resolver/`)

Project SQL never contains secret values. A `CREATE SECRET` body is a
provider expression resolved on the operator's machine at apply time —
`compile` never needs secret access:

- `env_var('NAME')` — read an environment variable;
- `aws_secret_manager('name')` — fetch from AWS Secrets Manager, using the
  `aws_profile` configured on the active mz-deploy profile;
- `json_field(<provider>, 'key')` — extract a field from a JSON secret.

`SecretResolver` rewrites the statement's body to a string literal just
before execution. Connections that reference secrets get the same treatment
when their options are applied. Expressions that are not recognized
provider calls pass through unchanged, so plain literal secrets still work.

Where secret values can and cannot appear: they exist only in memory on the
operator's machine and in the DDL sent to the server. The compiler cache
stores the unresolved provider expression, not the value; plan output and
`--output json` carry the parallel redacted statements the executor
maintains; and AWS access uses the standard credential chain under the
profile's configured `aws_profile`.

#### setup (`cli/commands/setup.rs`)

`setup` is the once-per-environment bootstrap that everything else verifies
(`setup::verify()`) before touching metadata:

1. Create the `_mz_deploy_server` cluster — the dedicated compute that
   every mz-deploy connection pins its session to, keeping metadata queries
   off user clusters.
2. Create the `_mz_deploy` database and execute `SETUP_STATEMENTS`
   (`setup_schema.rs`): the `tables.*` metadata tables, their indexes, and
   the `public.*` convenience views documented in
   [server-side state](#server-side-state).
3. On RBAC-enabled environments, create the `materialize_deployer`,
   `materialize_developer`, and `materialize_monitor` roles with their
   grants. Profiles marked `emulator = true` skip role checks entirely.

Every statement is `IF NOT EXISTS`-guarded or pre-checked, so rerunning
`setup` is safe and is also the upgrade path when the metadata schema gains
tables (the `tables.version` row records the installed version).

#### delete and clean

`delete` (`cli/commands/delete.rs`) is the explicit destructive counterpart
to apply: it drops a named infrastructure object after confirmation and
removes its definition file from the project, keeping disk and server in
agreement. `clean` (`cli/commands/clean.rs`) is purely local — it removes
the compiler cache directory.

### Unit testing

`mz-deploy test` runs unit tests for views and materialized views against a
real Materialize — a local Docker container — with every dependency
replaced by inline mock data. No remote environment, no project deployment:
the test harness lowers each test into a handful of temporary views plus
one assertion query, runs them, and reports the row-level difference
between expected and actual output.

#### Test definition

Tests live inline in the same `.sql` file as the object they test:

```sql
EXECUTE UNIT TEST test_name
FOR db.schema.view_name
[AT TIME '<expression castable to mz_timestamp>']
MOCK db.schema.dependency (col1 type, col2 type) AS (
  VALUES (...), (...)
)
EXPECTED (col1 type, col2 type) AS (
  VALUES (...)
);
```

`EXECUTE UNIT TEST` is parsed by `mz_sql_parser` like any other statement.
The compiler extracts these statements onto their owning `DatabaseObject`,
and dependency analysis converts them to the IR in
`project/ir/unit_test.rs` — `UnitTest { name, target_view, at_time, mocks,
expected }`, collected on `graph::Project::tests` as `(ObjectId, UnitTest)`
pairs. Because tests ride through the normal compile pipeline, they are
cached, recompiled incrementally, and visible to the LSP (which renders a
"run test" code lens on each one).

#### Command flow (`cli/commands/test.rs`)

1. **Compile** the project (`project::plan()`), optionally with an overlay
   filesystem (`--overlay <json>`, a JSON object mapping absolute file
   paths to contents that shadows those files on disk) carrying unsaved
   editor buffers — this is how the VS Code extension runs tests against
   dirty files.
2. **Load types.** Column schemas come from the typecheck cache
   (`ProjectCache`) for in-project objects, falling back to `types.lock`
   for external dependencies. Both are needed by validation below; if
   neither is available the test fails with a pointer to run
   `mz-deploy lock`.
3. **Connect** to the local container via `DockerRuntime::get_client()`.
4. **Filter.** The positional filter matches
   `database.schema.object#test_name` with `*` wildcards per segment.
5. **Run each test** (`run_single_test()`), classify the outcome, print a
   cargo-test-style line per test, a summary, and optionally `--junit-xml`
   output (one suite per target object).

Exit code 0 only if every selected test passed.

#### Validation before execution

`lower.rs::validate_unit_test()` rejects bad tests before any Docker work:

- **Unmocked dependency** — every *direct* dependency of the target view
  must have a mock. Direct is enough: lowering rewrites the target's own
  references to point at mocks, so transitive dependencies never execute —
  which is also why `test` needs none of `explain`'s recursive staging.
  (Conversely, mocks fully determine the input universe; tests cannot read
  real data.)
- **Mock schema mismatch** — each mock's declared columns must match the
  real object's column names and types.
- **Expected schema mismatch** — the `EXPECTED` columns must match the
  target view's typechecked output schema.

Type comparison normalizes aliases (`int`/`int4`/`integer`,
`text`/`varchar`, bare `list` vs `int8 list`) so cosmetic type spellings
don't fail tests. Errors render with column-level suggestions.
`validate_at_time()` separately checks that the `AT TIME` expression casts
to `mz_timestamp`.

#### Lowering (`cli/commands/test/lower.rs`)

`lower_unit_test()` emits an ordered list of SQL statements:

1. **One temporary view per mock**, named with the *flattened* form of the
   dependency's FQN (a single quoted identifier like `"db.schema.users"`):

   ```sql
   CREATE TEMPORARY VIEW "db.schema.users" AS
   WITH MUTUALLY RECURSIVE data(id bigint, name text) AS (
     VALUES (1, 'alice')
   )
   SELECT * FROM data;
   ```

   The `WITH MUTUALLY RECURSIVE data(<typed columns>)` wrapper is
   load-bearing. An ordinary CTE infers its column types from its body. A
   recursive CTE *declares* its column names and types, so the mock's rows
   are coerced to the real object's schema regardless of what the `VALUES`
   literals would infer.

2. **A temporary `expected` view**, same wrapper, from the `EXPECTED`
   block.

3. **The target view, rewritten.** The target's real CREATE statement is
   re-parsed and run through `NormalizingVisitor::flattening()`, which
   rewrites every dependency reference to the flattened mock name — so the
   production SQL text executes verbatim, except that each `FROM`/`JOIN`
   target now resolves to a mock. The result becomes a
   `CREATE TEMPORARY VIEW "db.schema.target"`.

4. **The assertion query**: a symmetric difference labeling each
   discrepancy,

   ```sql
   SELECT 'MISSING', *    FROM expected EXCEPT SELECT 'MISSING', *    FROM target
   UNION ALL
   SELECT 'UNEXPECTED', * FROM target   EXCEPT SELECT 'UNEXPECTED', * FROM expected
   [AS OF <at_time>::mz_timestamp]
   ```

   Zero rows is a pass. Otherwise rows are split into MISSING (expected but
   absent) and UNEXPECTED (present but not expected) and rendered as a
   table. The optional `AS OF` makes `mz_now()`-dependent views testable at
   a fixed timestamp.

All views are temporary, so per-test cleanup is a single `DISCARD ALL` — no
state survives between tests even within one session.

#### The Docker runtime (`docker_runtime.rs`)

Tests (and `explain`) run against a long-lived local container named
`mz-deploy-typecheck`, publishing Materialize on host port 16875, started
with `MZ_EAT_MY_DATA=1` and raised object limits. `get_client()` tries a
fast-path connection first and only shells out to Docker (existence check →
running check → health check → restart or recreate) when that fails, so the
steady-state cost of `mz-deploy test` is one connection attempt. The image
comes from `project.toml`'s `mz_version` or the `--docker-image` flag.

Two sharp edges: the container is shared *globally*, not per project — any
project on the machine reuses it; and reuse is by name, not image, so
changing the image requires `docker rm -f mz-deploy-typecheck` before the
new one takes effect. Connections to it use the unpinned variant
(`connect_with_profile_no_pin`) since the container has no
`_mz_deploy_server` cluster.

#### Outcome model

```
TestOutcome::Passed
TestOutcome::ValidationFailed   (schema/AT TIME problems, caught pre-Docker)
TestOutcome::Failed(Error)      (a setup statement or query errored; the report
                                 says which statement — mock N, expected, target)
TestOutcome::Failed(AssertionFailed { columns, missing, unexpected })
```

There is deliberately no snapshot-update mode: expected results are
hand-written and changed by editing the test, so behavior changes always
show up in review as a diff to the `EXPECTED` block.

#### CI integration

`test/mz-deploy/mzcompose.py` (repo root) exercises the whole flow against
fixture projects under `test/mz-deploy/projects/`, asserting on test names
in output and on exit codes — including the intentional-failure path. All
workflows there run against the emulator and are CI-safe.

### explain (`cli/commands/explain.rs`)

`mz-deploy explain db.schema.object[#index]` shows the optimizer plan for a
materialized view or index *before it is ever deployed*. Like `test`, it
runs against the local Docker container. The problem is reconstructing
enough of the target's dependencies for the optimizer to plan against.

The staging algorithm (`plan_staging` / `plan_dep`, a pure function over
the compiled graph) decides, per dependency, how to mock it. The question
for each dependency is: what about it can change the target's plan? For
anything the optimizer treats as an opaque input — a table, a materialized
view, an external object — only the column schema matters, so a bare stub
table suffices. Two things break that opacity. A plain view inlines into
the target's plan, so it must be created for real. An index **on the
target's cluster** changes which plans are available, so it must be
recreated. As rules:

```datalog
Need(D)        :- DirectDep(Target, D)                 % start from the target's deps
Need(D)        :- Inlined(V), DirectDep(V, D)          % recurse only through inlined views

StubTable(D)   :- Need(D), External(D)                 % external objects: schema only
StubTable(D)   :- Need(D), HasIndexOn(D, TargetCluster)
CreateIndex(I) :- Need(D), IndexOf(D, I), OnCluster(I, TargetCluster)
StubTable(D)   :- Need(D), NOT HasIndexOn(D, TargetCluster), Kind(D) ∈ {mv, table}
Inlined(D)     :- Need(D), NOT HasIndexOn(D, TargetCluster), Kind(D) = view
CreateView(D)  :- Inlined(D)                           % after its own deps are staged
```

The kind rules carry a `NOT HasIndexOn` guard, so the index rule
effectively takes precedence: even a *view* with an index on the target's
cluster is stubbed as a table plus its indexes rather than inlined — the
index makes it a materialized input from the optimizer's perspective.
Indexes on *other* clusters are ignored entirely; they cannot serve a query
running on the target's cluster and would only distort the plan.

An in-project dependency of any other kind errors with "cannot be staged
for explain". In practice that bites raw `CREATE SOURCE` objects: in the
project model, data is read through tables — including
`CREATE TABLE ... FROM SOURCE`, which has kind table and stubs cleanly — so
a view reading a source *directly* is the unsupported case. Stub column
types come from the typecheck cache, falling back to `types.lock`, falling
back to the dependency's own `CREATE TABLE` statement when it is a
project-defined table whose DDL declares its columns.

Everything is created in a uniquely named scratch schema
(`_mz_explain_<timestamp>`) with cluster references rewritten to the
container's `quickstart` cluster, the appropriate
`EXPLAIN MATERIALIZED VIEW` / `EXPLAIN INDEX` runs, and the output is
post-processed to substitute the user's real schema and cluster names back
in. The scratch schema is dropped in a finally-block. The `--overlay` flag
works here as in `test`, so the extension can explain unsaved edits.

### Inspection commands and utilities

Three thin read-only commands over `client/deployment_ops.rs` queries:

- **`list`** — active staging deployments (`promoted_at IS NULL`), each
  with deployer, age, commit, schemas, and a live per-cluster hydration
  readout using the same status logic as `wait`.
- **`describe <id>`** — one deployment in full: metadata, promotion status,
  and every object with its semantic hash.
- **`log`** — promotion history, newest first: who promoted what and when.

All three implement `Render`, so `--output json` produces stable structured
output.

And the utilities:

- **`sql`** (`cli/commands/sql.rs`) — translates the active profile into
  `PG*` environment variables and `exec`s `psql`, forwarding trailing
  arguments. It does *not* pin the session to `_mz_deploy_server`: an
  interactive shell should land on the user's default cluster.
- **`debug`** (`cli/commands/debug.rs`) — one-stop diagnostics: resolves
  the profile, connects, checks the `_mz_deploy_server` cluster's health
  (missing or unreplicated → suggests `setup`), and reports Docker
  availability for `test`/`explain`.
- **`profile`** (`cli/commands/profile.rs`) — `list`/`set`/`current` over
  the profile resolution chain (flag → env var → `.mzprofile` file). `set`
  writes the project-local `.mzprofile`.
- **`new` / `init`** (`cli/commands/new_project.rs`) — scaffold a project
  from templates embedded at compile time (`cli/scaffold/`):
  `project.toml`, `models/` skeleton, `.gitignore`, README, and a
  `.vscode/extensions.json` recommending the extension. On an interactive
  TTY it prompts to pick an initial profile.

#### The help system (`cli/extended_help.rs`)

Every command has a hand-written markdown page in `cli/help/*.md`, embedded
with `include_str!` and served by `mz-deploy help <cmd>` with alias
resolution (`build` → `compile`, `history` → `log`, ...). Each page follows
a fixed shape — summary, usage, numbered behavior description, flags, error
recovery, exit codes, related commands. `all_help()` concatenates every
page with delimiters, which is how the full command reference gets exported
into agent-facing contexts. When editing command behavior, the matching
help page is part of the change.

### Developer tooling: LSP, MCP, VS Code

#### LSP server (`lsp/`)

`mz-deploy lsp` serves the language protocol over stdio via `tower-lsp`
(`lsp/run.rs`, `lsp/server.rs`). The design premise is that the LSP is a
*reuse* of the compiler, not a second implementation: every capability is
answered from the same `graph::Project` IR and typecheck cache the CLI
builds.

The `Backend` keeps open documents as ropes, plus two diagnostic layers:
per-keystroke parse diagnostics (cheap, published immediately) and
project-level diagnostics from a full `project::plan_sync()` + typecheck
rebuild, debounced 100ms after the last edit and serialized behind a
rebuild lock. Open buffers feed the rebuild through the same overlay
filesystem mechanism `test --overlay` uses. A watched-files handler catches
external edits to `project.toml` and `.mzprofile`.

Implemented capabilities: completion, hover, goto-definition, references,
document symbols, workspace symbols, code lenses, code actions, and
semantic tokens (full-document sync). Completion (`lsp/completion.rs`)
resolves the file's default database/schema from its path, then offers
functions (from `mz_sql::func`), keywords, project and external objects —
and column completions *only for objects that are dependencies of the
current file*, resolving table aliases to object ids. Code lenses surface
`mz-deploy.runTest` and `mz-deploy.runExplain` commands on unit tests and
materialized views, which the extension wires to CLI invocations.

#### MCP proxy (`cli/commands/mcp.rs`)

`mz-deploy mcp` is a small stdio↔HTTP bridge so MCP clients (Claude Code,
Claude Desktop, Cursor) can reach a Materialize environment's developer MCP
endpoint: it reads newline-delimited JSON-RPC on stdin, POSTs each request
to `<http_host>/api/mcp/developer` with basic auth from the profile, and
writes responses to stdout. It resolves a profile without requiring
`project.toml`, infers the URL scheme (https for remote hosts, http for
loopback, default port 6876), and converts HTTP failures into synthesized
JSON-RPC errors so clients see structured failures rather than hangs.

#### VS Code extension (`misc/vscode-ext/`)

The extension is deliberately thin: it spawns `mz-deploy lsp -d <workspace>`
as the language server and otherwise *shells out to the CLI* rather than
inventing custom protocol methods. Code-lens commands snapshot dirty
buffers into a temporary overlay JSON file and run `mz-deploy test <filter>
--overlay <file>` or `mz-deploy explain <target> --overlay <file>` as VS
Code tasks, deleting the overlay afterward. A status-bar profile switcher
wraps `mz-deploy profile list --output json` / `profile set`, and watches
`.mzprofile` for external changes. The binary path comes from the
`mz-deploy.path` setting (default: `mz-deploy` on PATH), validated with
`--version` at activation.

The extension adds no behavior of its own. Anything reachable from the
editor is reachable, identically, from a terminal or CI.

### Output, errors, and diagnostics

`log.rs` routes all human-facing output: `info!`/`verbose!` macros write to
stderr, structured results implement `Render` and go through
`log::output()`, which serializes to stdout as JSON under `--output json`
and pretty-prints to stderr otherwise. Commands print machine-consumable
identifiers (like a deploy id) on bare stdout only in text mode.

Compile-time errors flow through `diagnostics.rs`, which converts typed
errors (`ProjectError`, typecheck failures, validation failures) into
`PositionalDiagnostic`s — file, byte range into the original source,
message, help footers, and rustc-style suggestions — rendered with
`annotate_snippets`. Byte-offset bookkeeping in the parser
(`project/syntax/`) is what allows an error in variable-substituted SQL to
point back at the user's original file.

### Where the complexity is

For reviewers triaging where to spend attention, the largest and most
consequential modules are:

- `project/compiler/typecheck/catalog.rs` (~2.5k lines) — the offline stub
  catalog that typechecks SQL without a server.
- `cli/commands/stage.rs` (~2k lines) — changeset computation,
  partitioning, staging resource creation with rollback.
- `cli/commands/test/lower.rs` (~2k lines) — rewriting user views into
  temporary mocked views and assertion queries.
- `client/deployment_ops.rs` (~1.6k lines) — every read/write of the
  `_mz_deploy` metadata schema, including the crash-recovery state machine.
- `cli/commands/promote.rs` (~1.1k lines) — the atomic swap transaction and
  resumable post-swap work.

## Minimal Viable Prototype

The full implementation exists and is validated end to end:

- The `mz-deploy` crate (`src/mz-deploy/`, ~59k lines) implements every
  command described above.
- `test/mz-deploy/mzcompose.py` exercises the complete lifecycle — setup,
  apply, stage, wait, promote, abort, dev, unit tests including the
  intentional-failure path — against fixture projects in
  `test/mz-deploy/projects/`, running on the emulator so all workflows are
  CI-safe.
- The crate carries in-crate unit tests (e.g. ~2.4k lines covering
  normalization alone), and the unit test harness, LSP, and VS Code
  extension have been used to develop real projects against both the
  emulator and cloud environments.

## Alternatives

- **Keep investing in dbt-materialize.** The strategic case against this
  is made in [The Problem](#the-problem): `dbt build` is wrong for an
  operational platform, the vendor is consolidating around warehousing,
  there is no onboarding timeline for dbt Fusion or dbt Cloud, and we
  would continue not to own our product's primary interface. There is
  also a technical ceiling no adapter work can fix. dbt's model is
  templated-SQL-in, rendered-SQL-out. It cannot typecheck against
  Materialize's planner offline. It has no notion of
  clusters/sinks/secrets as managed objects. It cannot express the atomic
  multi-schema-and-cluster swap, partial deployments, replacement views,
  or crash-resumable promotion. Its tests run against warehouse data, not
  inline mocks. dbt-materialize continues to exist for teams already
  standardized on dbt, but it cannot be the foundation.
- **Extend the Terraform provider.** Terraform's resource model fits the
  `apply` half of the problem: idempotent reconciliation of clusters,
  connections, secrets. It does not fit the rest. There is no useful
  resource encoding for views, materialized views, indexes, blue/green
  staging, hydration-gated cutover, deferred sinks, or
  `APPLY REPLACEMENT`. Its state file duplicates — and can drift from —
  the catalog; mz-deploy keeps all state in the server itself.
- **Server-side deployment primitives only.** One could push the whole
  lifecycle into the server (a `DEPLOY` statement family). Instead, the
  server-side pieces this design needed — `SWAP WITH`,
  `APPLY REPLACEMENT`, `ALTER SINK ... SET FROM` — were built as
  composable primitives. Policy (changesets, partitioning, conflict
  semantics, project layout) stays in a client that can iterate quickly
  and run offline work (compile, typecheck, test, explain) with no server
  at all.
- **Migration-script tooling (Flyway/Liquibase style).** Ordered migration
  scripts record *how to get here*, not *what should exist*. They cannot
  diff against live state or compute a minimal blue/green changeset, and
  they put the burden of writing correct forward/backward DDL on the
  user. The declarative model was chosen so the tool, not the user,
  derives the DDL.

## Open Questions

- **Metadata schema migrations.** `tables.version` is recorded but nothing
  reads it yet; the upgrade path is "rerun `setup`". A future version that
  changes (rather than adds) tables will need real migration machinery and
  a compatibility check.
- **The `Tables` deployment kind.** `DeploymentKind::Tables` exists in
  `client/models.rs` but no command writes it. It should either gain a
  writer (recording `apply tables` runs as deployments) or be removed.
- **Dev overlay naming.** Overlay databases are named
  `<database>__<profile>` while the manifest is keyed by (profile,
  project); two projects declaring the same database name under one profile
  would collide on the overlay name. Worth either a project component in
  the name or a documented restriction.
- **Docker image rotation.** The shared `mz-deploy-typecheck` container is
  reused by name, so changing `mz_version` requires a manual
  `docker rm -f`. The runtime could compare the running container's image
  against the requested one and recreate automatically.
- **`lock` against a moving server.** `types.lock` staleness is detected
  only at deploy time. A cheap server-side check (digest of external
  dependencies' columns) during `stage` could surface drift earlier.
