# catalog::durable::upgrade::v86_to_v87

Migration from catalog version 86 to 87.

Adds durable cluster-autoscaling state and reshapes the managed replica location's availability-zone representation. Two record shapes change between v86 and v87:

- **`ManagedCluster`** gains three optional fields: `auto_scaling_strategy` (the user-configured autoscaling policy), `reconfiguration` (an in-flight graceful reconfiguration the controller is converging on), and `burst` (an in-flight hydration burst the controller is running). All three default to `None` for existing clusters.
- **Managed `ReplicaLocation`** replaces the single optional `availability_zone` user-pin with an `availability_zones` list. For a replica belonging to a managed cluster the list is backfilled from the owning cluster's `availability_zones` pool. For a replica of an unmanaged cluster the prior single-AZ pin is carried across as a zero- or one-element list.

Unmanaged clusters and unmanaged replica locations are JSON-identical between v86 and v87 and are left untouched; emitting an update for them would retract and re-add the same bytes with no effect.

## Migration strategy

The `upgrade` function takes a full v86 state snapshot and returns a list of `MigrationAction::Update` pairs:

1. **Collect cluster AZ pools**: scan the snapshot for managed `Cluster` records and build a `BTreeMap<ClusterId, Vec<String>>` mapping each managed cluster to its `availability_zones`.
2. **Rewrite managed clusters**: for each managed `Cluster`, reconstruct the record with the three new fields set to `None` and emit an `Update` from the v86 form to the v87 form. Unchanged sub-structures are moved via `JsonCompatible::convert`.
3. **Rewrite managed replica locations**: for each `ClusterReplica` with a `Managed` location, look up the owning cluster's AZ pool from the map. If the cluster is managed, use that pool as the `availability_zones` list; if unmanaged (no map entry), carry the replica's prior `availability_zone` pin as a zero- or one-element list. Emit an `Update` pair.

Sub-structures that are unchanged between the two versions (`ClusterKey`, `RoleId`, `MzAclItem`, `ReplicaLogging`, `OptimizerFeatureOverride`, `ClusterSchedule`, `ClusterReplicaKey`, `ClusterId`, `UnmanagedLocation`) are declared JSON-compatible via the `json_compatible!` macro, which also generates proptest round-trips asserting byte-level compatibility.
