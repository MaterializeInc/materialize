{{- /* Describes how a Materialize source behaves when the upstream table undergoes schema changes or destructive operations. Single source of truth shared by the Postgres, MySQL, and SQL Server ingestion pages. Takes a `connector` param (`postgres`, `mysql`, or `sql-server`). Source- versioning links are built from the connector so each page links to its own guide. Postgres-only concepts (publications, replica identity, exclusion constraints) render only when connector == "postgres". Call with the `{{% %}}` form so the inner markdown renders: {{% upstream-schema-change-behavior connector="postgres" %}} */ -}} {{- $connector := .Get "connector" -}} {{- $is_postgres := eq $connector "postgres" -}} ### Adding a column By default, when you add a new column to your upstream database, nothing changes in Materialize. Materialize continues to ingest the existing columns. To incorporate the new column, create a new table from the source. See [Handle upstream column addition](/ingest-data/{{ $connector }}/source-versioning/#handle-upstream-column-addition). ### Dropping a column If your Materialize source ingests a column, dropping that column from your upstream database causes ingestion to stall. You can safely drop a column by first ignoring it in Materialize. See [Handle upstream column drop](/ingest-data/{{ $connector }}/source-versioning/#handle-upstream-column-drop). ### Changing constraints Materialize ignores foreign key and `CHECK`{{ if $is_postgres }} and `EXCLUSION`{{ end }} constraints today, so you can change them with no impact to Materialize. Adding a new `UNIQUE` or `NOT NULL` constraint also has no impact. Dropping a `UNIQUE` or `NOT NULL` constraint isn't supported today. ### Operations that require recreating the table The following upstream operations put the affected table into an error state. Ingestion for that table stops, and you must drop and recreate the affected table in Materialize to resume: - Dropping a table (`DROP TABLE`){{ if $is_postgres }}, removing it from the publication (`ALTER PUBLICATION ... DROP TABLE`), or dropping the publication (`DROP PUBLICATION`){{ end }} - Renaming a column - Renaming a table or moving it to a different schema - Changing a column's data type (except changing it to `text`) {{ if $is_postgres }}- Setting a table's replica identity to anything other than `FULL` (`ALTER TABLE ... REPLICA IDENTITY`) {{ end }}{{ if ne $connector "sql-server" }}- Truncating a table (`TRUNCATE`) {{ end }}
